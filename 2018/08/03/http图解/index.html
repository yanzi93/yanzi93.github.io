http图解

发表于 2018-08-03 | 分类于 http

浅读《图解HTTP》，这本书有一点老了，信息跟不上现在更新的步伐。不过作为了解的入门还是非常浅显易懂的。这也算不上什么笔记，只是对书中认为重要的知识点做了一个小的概括。

web及网络基础

TCP/IP

把与互联网关联的协议结合起来称为TCP/IP。

TCP/IP分层

TCP/IP是分层的，进行网络通信时发送端从应用层往下走，接收端则往应用层上走。

  • 应用层 FTP、DNS、HTTP

    决定了向用户提供应用服务时通信的活动。

  • 传输层 TCP、UDP

    对上层应用层，提供处于网络连接中两台计算机之间的数据传输。

    进行数据分割，并在报文上标记序号及端口号

  • 网络层 IP

    处理在网络上流动的数据包，规定传输路线。

    增加MAC地址转发给链路层

  • 链路层

    处理连接网络的硬件部分。硬件上的范畴均在链路层的作用范围。

IP

位于网络层，作用就是把各种数据包传送给对方。

IP地址指明节点被分配地址，可以和MAC地址进配对。

TCP

位于传输层，提供可靠的字节流服务。及把大数据分割为报文并准确送达对方。

TCP如何确保准确送达目的地：

TCP三次握手

  1. 发送端发送一个带SYN标志数据包给对方
  2. 接受端收到后回传一个SYN/ACK标志数据包确认信息
  3. 发送端回传一个带ACK标志数据包，代表握手结束建立连接。

TCP握手阶段莫名中断，会以相同顺序放相同数据包。

DNS

Domain Name System，位于应用层，提供域名到IP地址之间的解析服务或逆向查询。

URI、URL

URI：统一资源标识符，用于标识互联网资源

URL：表示资源的地点。是URI的子集。

浏览器打开页面过程

  1. 在地址栏输入URL回车
  2. 发送给DNS域名，查询对应IP地址。
  3. HTTP协议生成针对WEB服务器的HTTP请求报文
  4. TCP将请求报文分为报文段发送
  5. IP协议负责网络传送线路
  6. 接收方TCP接收到报文段重组位报文
  7. 接收方HTTP对web服务器请求内容进行处理

简单HTTP协议

http协议默认80端口；https默认使用443端口

http：HyperText Transfeer Protocol，超文本传输协议。

特点

  • 无状态：不做持久化处理。

    解决无状态引入cookie技术

  • 持久连接 HTTP1.1开始默认持久连接

  • 请求必须由客服端发出，服务端回复响应。

缺点

  • 通信使用明文 窃听
  • 不验证通信方身份 伪装
  • 无法证明报文完整性 篡改

HTTP方法

  • GET获取资源
  • POST传输实体主体
  • PUT传输文件，不带验证机制存在安全问题
  • HEAD获得报文首部，用于确定URI有效性及资源更新的日期等
  • DELETE删除文件，不带验证机制
  • OPTIONS询问支持方法
  • TRACE追踪路径，容易引发XST跨站追踪攻击
  • CONNECT要求用隧道协议连接代理，主要使用SSL、TLS

持久连接 （HTTP1.1默认持久连接）

只要任意一端没有明确提出断开连接，则保持TCP连接状态。

好处：减少TCP连接的重复建立和断开造成的额外开销，减轻服务器荷载。提高web页面显示速度。

管线化

不用等待响应亦可发送下一个请求。并发发送请求。

Cookie状态管理

通过在请求和响应报文中写入Cookie信息控制客户端状态。

HTTP信息

HTTP报文大致分为：报文首部+ 空行+报文主体。

请求报文

报文首部 + 空行 + 报文主体

  • 报文首部：请求行+请求首部字段+通用首部字段+实体首部字段+其他
  • 请求行：请求方法，请求URI，协议版本

响应报文

报文首部 + 空行 + 报文主体

  • 报文首部： 状态行+响应首部字段+通用首部字段+实体首部字段+其他
  • 状态行： 协议版本，状态吗，解释状态码的原因短语

Cookie相关首部字段

  • set-Cookie 响应首部字段

    expires：有效期（默认浏览器关闭前为止）

    domain：域名（默认创建Cookie的服务器域名）

    HttpOnly：加以限制，使Cookie不能被js脚本访问，防止跨站脚本攻击（XSS）

    Secure：仅在HTTPS安全通信发送Cookie

  • Cookie 请求首部字段

发送方式和格式

  1. 压缩传输内容编码

  2. 分割发送的分块传输编码

  3. 发送多种数据的多部分对象集合：MIME

    在HTTP首部字段加Content-type

    • multipart/form-data：web表单文件上传
    • multipart/byteranges：状态码206相应报文包含多个范围的内容

  4. 获取部分内容的范围请求：

    请求头部加Range，并且首部字段Content-Type标明multipart/byteranges

    响应返回状态码206，无法响应范围请求则返回全部和200

  5. 内容协商返回最合适内容（Content Negotiation）

媒体类型

  • 文本文件
    • text/html, text/plain, text/css
    • application/xhtml+xml, application/xml
  • 图片文件
    • image/jpeg, image/gif, image/png
  • 视频文件
    • video/mpeg, video/quicktime
  • 应用程序使用的二进制文件
    • application/octet-stream, application/zip

HTTP状态码

负责表示客服端HTTP请求的返回结果，标记服务器端的处理是否正常，通知出现的错误等工作。

状态码以3位数字和原因短语组成，第一位指定响应类别。

状态码类别

  • 1XX 接受的请求正在处理 信息性状态码
  • 2XX 请求正常处理完毕 成功状态码
    • 200 OK
    • 204 No Content 返回响应报文不含实体的主体部分
    • 206 Partial Content 范围请求
  • 3XX 需要进行附加操作以完成请求 重定向状态码
    • 301 Moved Permanently 永久性重定向
    • 302 Found 临时重定向 （如保存在书签中，不会修改书签保存地址） POST变成GET
    • 304 Not Modified 客服端发送附带条件请求时服务端逊于请求访问资源单未满足条件（返回不包含任何响应主体部分）。
    • 307 Temporary Redirect 临时重定向。禁止POST变成GET
  • 4XX 无腹鳍无法处理请求 客户端错误状态码
    • 400 Bad Request 请求报文存在语法错误
    • 401 Unauthorized 需要认证信息或认证失败
    • 403 Forbidden 请求被拒绝，没有资源访问权
    • 404 Not Found 未找到资源
  • 5XX 服务器处理请求出错 服务器错误状态码
    • 500 Internal Server Error 服务端执行请求发生错误
    • 503 Service Unavailable 服务器暂时处于超负荷或停机维护

Web服务器

使用单台虚拟主机实现多个域名：（Web托管服务）IP地址相同，根据域名host区分。

通信数据转发程序：代理、网关、隧道

将请求转发给通信线路上的下一站服务器，并且能接受从那台服务器发送的响应再转发给客户端。

代理

有转发功能的应用程序，扮演服务器和客户端中间人角色为两者传递信息。

  • 代理不改变请求URI，会直接发送给前方目标服务器。
  • 每次通过代理服务器，会追加Via首部信息

使用代理原因：利用缓存技术减少网络带宽的流量，组织内部针对特定网站的访问控制，以获取访问日志位主要目的等。

代理方法：缓存代理，透明代理

网关

转发其他服务器通信数据的服务器。

  • 利用网关可以由HTTP请求转化为其他协议通信（区别于代理）
  • 提高通信的安全性。客户端和网关之间可以加密

隧道

在客户端和服务端之间进行中转，并保持双发通信连接的应用程序。使用SSL等加密手段进行通信。

目的：确保客户端能与服务器进行安全的通信，隧道本身不会解析HTTP请求。

缓存

指代理服务器或客户端本地磁盘保存的资源副本。

优点：减少对资源服务器的访问，节省了通信流量和通信时间。

缓存服务器优点：避免多次从原服务器转发资源，客户端就近获取资源。

HTTPS

解决http存在部分缺陷，在http上加入加密处理和认证机制即https。HTTPS采用混合加密机制（共享密钥和公开密钥加密混用）。

与HTTP区别

通信接口部分使用SSL和TLS协议代替。通常HTTP直接和TCP通信，改为先和SSL通信，在再由SSL和TCP通信。使HTTPS拥有加密，证书和完整性保护功能。

为什么不一致使用HTTPS

  1. 加密通信消耗更多CPU及内存资源
  2. 购买证书需要开销

SSL

加密方式

采用公开密钥加密，公开密钥加密使用非对称密钥（私有密钥，公开密钥）。

发送密文的一方使用对方的公开密钥进行加密处理，对方收到加密信息后使用自己的私有密钥进行解密。

如何证明公开密钥的正确性：数字证书

效率

处理速度变慢。通信慢并大量消耗CPU即内存等资源。

解决方案：使用SSL加速器硬件改善。

用户身份认证

BASIC认证

需要认证，服务器返回状态吗401

放松明文密码

DIGEST认证

质询响应，等级高于BASIC认证

SSL客户端认证

证书认证，需要费用。

表单认证

Session管理及Cookie应用

  • 客服端把用户ID和密码等登陆信息放入报文实体，以POST发送

  • 服务器发放用以识别用户的Session ID。通过验证从客户端发送来的登陆信息进行身份认证，把用户认证状态与Session ID绑定记录在服务端。

    客户端返回响应时写入Set-Cookie

  • 客户端接受从服务器发来的Session ID，将其作为Cookie保存在本地。再次自动携带Cookie发送请求。

基于HTTP的功能追加协议

Ajax

异步js与XML技术，有效利用js和DOM操作以达到局部Web页面替换加载的一部通信手段。只更新部分页面，传输数据量更小。

核心：XMLHttpRequest的API

缺点：利用Ajax实时从服务器获取内通，会导致大量请求产生；未解决HTTP协议本身存在问题；破坏浏览器前进后退功能。

SPDY

Google 2010年发布

  • 多路复用，单一TCP连接可处理多个HTTP请求
  • 赋予请求优先级
  • 压缩HTTP首部
  • 推送功能，支持服务器主动向客户端推送数据功能
  • 服务器提示功能，主动提示客户端请求所需资源

缺点：只用单个域名通信多路复用

2016年结束支持，为HTTP2.0前身

WebSocket

单个TCP连接上建立的全双工通信

建立在HTTP基础上的协议，连接放发起方仍是客户端，一旦建立不论服务器还是客户端均能直接向对方放松报文。

  • 推送功能

  • 减少通信量

    握手请求：

    Upgrade: websocket Connection:Upgrade

    握手响应：

    状态吗101 Switching Prtocols

Web攻击技术

  • XSS跨站脚本攻击

    通过存在安全漏洞的Web网站注册用户的浏览器內运行非法的HTML标签或者js进行攻击。

  • SQL注入攻击

    针对Web应用使用的数据库，通过运行非法SQL产生的攻击。

  • OS命令注入攻击

    通过Web应用执行非法的造作系统命令达到攻击的目的。

  • 会话劫持

    通过某种手段拿到用户会话ID，伪装用户
